## Not fixed (recommendations)
- **Stop latency**: stop is honored between batches; with 4 concurrent multi-minute local vision
  calls it can take minutes to actually halt. Fine by design, but the UI could show "stopping…".
- **/api/token** hands the bearer token to any loopback caller (deliberate bootstrap; fine for a
  single-user box, documented here for awareness).
- **Scan runs synchronously in the request** — minutes-long POST for 26k files; consider making it
  a job like indexing.
- **No person management UI** (rename/delete a registered person; person_map.json is edit-by-hand).
- **Thumb pregeneration**: first visit to a grid generates thumbs on demand (visible blank tiles
  for a few seconds). A post-scan thumbnail job would smooth this.
- **folders.json image_count** counts files walked, not cataloged (26,577 vs 24,764) — confusing label.
- Design polish ideas: lightbox ✕ overlaps caption text at narrow widths; pipeline cards could
  deep-link to sections; occasional free-form attribute values from the model ("party") could be
  normalized to the prompt's enum at parse time.

## Scale notes (measured at 24.7k images)
- status 0.15 s warm / timeline 0.02 s / recent 0.01 s / filters 0.07 s after fixes.
- PhotoGrid uses content-visibility:auto — large grids stay smooth; timeline now pages.
- ChromaDB queries fine at this size; the catalog is one 24k-entry JSON (~30 MB) rewritten
  per SAVE_EVERY=25 during jobs — acceptable; consider SQLite if the library goes 100k+.
- Local vision throughput ≈ 1.5-2 captions/min (gemma-4-e4b, conc 4). Full library ≈ days of
  GPU time; Gemini fallback is faster but rate-limited. Both resumable (per-model caption history).
